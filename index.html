Circuit breaker — fast failure beats a 30s timeout the caller hears as silence.

async def call(self, fn):
    if self.state == "OPEN" and now()-self.opened < self.recover:
        raise BreakerOpen()
    try:
        r = await fn(); self.fails = 0; self.state = "CLOSED"; return r
    except Exception:
        self.fails += 1
        if self.fails >= self.threshold:
            self.state = "OPEN"; self.opened = now()
        raise

Problem

Tier-2 hospitals have intermittent power and internet. Most software treats offline as a bug; a dropped connection mid-consultation loses the record.

What it does

Patient care continues fully offline — every mutation is written ahead and queued locally, then synced when connectivity returns. Nothing is lost to a crash or outage.

Who uses it

Doctors and front-desk staff on tablets, in buildings where the network is not a given.

System

Durability
WAL: PENDING → COMMITTED → APPLIED before the row mutation
Queue
MySQL SELECT … FOR UPDATE SKIP LOCKED · multi-worker safe
Identity
Client-generated UUIDv7 idempotency registry
Offline

Access
9-role RBAC · immutable audit log · Zod types shared client/server

Key decisions

WAL before DBCOMMITTED-but-not-APPLIED replays on recovery; PENDING discards.
SKIP LOCKEDMySQL gives durable queue semantics without a second system.
Shared Zod typesOne schema, no client/server drift.

Failure modes & handling

  • Crash mid-write — WAL invariant: committed intent replays, partial pending discards.
  • No connectivity — all mutations to IndexedDB first; UI optimistic; sync on reconnect.
  • Sync retry storms — UUIDv7 idempotency registry dedupes replayed mutations.
  • Concurrent workers — SKIP LOCKED prevents double-processing a job row.

Evidence

WAL recoveryCorrect crash-recovery semantics, the hard part most skip.
9-role RBACReal authorization model + immutable audit log.
